Phase 6 — Observability UI

Goal: Surface every participant's activity in its own panel. Each agent/tool gets a dedicated place in the UI with an activity log and settings controls. Critic (currently invisible) gets a panel. The generator's episodic memory becomes browsable via a memory inspector.

Motivation

Phase 5 introduced a shadow generator (RespondentB), pairwise grading, and engram annotations — but none of it is observable from the UI. You can't see what the critic scored, what's in memory, or whether the pairwise winner annotation landed. Phase 6 makes all of that visible.

The existing tool panel structure (Activity + Settings per tool) is the right pattern — extend it to cover Critic and add a new Memory Inspector panel. No new interaction patterns needed; just more panels.

New Panels

CriticPanel src/local/ui/critic_panel.py

Same two-tab structure as ToolPanel. No new bus subscription needed — it connects to BusLogger.critique which already exists and is emitted on every critique.result event.

Activity tab

Each grade event appended as a compact card:

[21:07:04.3]  ● 4/5  (pairwise: A won)
  Q: What is the boiling point of water?
  Feedback: Accurate and concise. Minor omission…
  • Score badge colour-coded (same scheme as conversation panel)
  • Shows respondent_id and pairwise_winner when present
  • Feedback text truncated to ~120 chars, full text on hover (tooltip)

Settings tab

Reuses ToolPanel's YAML editor + Save pattern for config/critic.yaml. Note: critic saves do not trigger TOOL_SCHEMA_REQUEST (critic has no tool schema). Save writes the file only; critic picks it up on next restart.

MemoryInspectorPanel src/local/ui/memory_inspector.py

Pull model — no bus subscription. Queries MemoryService directly on Refresh. Safe to call from Qt thread (ChromaDB reads are thread-safe).

BusLogger — pairwise enrichment

critique.result payloads currently carry respondent_id and score but not pairwise_winner (that lands later via pairwise.result). The simplest approach for Phase 6: CriticPanel also subscribes to PAIRWISE_RESULT via BusLogger. When a pairwise.result arrives, append a separate compact line to the activity log: 

[21:07:18.1]  ⟺ pairwise  winner=A
  A: 46868de7  B: 8a1c2f39

No need to retroactively update the absolute grade cards — the pairwise line lands a few seconds later and the sequence is self-explanatory.

MemoryService — list_episodic()

def list_episodic(self, n: int = 100) -> list[dict]:
    """Return the n most recent episodic engrams, newest first."""
    result = self._collection.get(
        where={"type": "episodic"},
        include=["metadatas", "documents"],
    )
    ids   = result.get("ids") or []
    docs  = result.get("documents") or []
    metas = result.get("metadatas") or []
    items = sorted(
        zip(ids, docs, metas),
        key=lambda x: x[2].get("timestamp", 0),
        reverse=True,
    )
    return [
        {"id": id_, "content": doc, "metadata": meta}
        for id_, doc, meta in items[:n]
    ]
ChromaDB get() returns all matching documents. For typical LoCAL2 usage (hundreds of engrams) this is fast. If the collection grows very large (thousands), add a timestamp filter. For Phase 6, the simple approach is fine.
