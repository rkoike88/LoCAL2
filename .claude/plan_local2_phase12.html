Phase 12 — Multi-Collection RAG Library

Extends the single-topic document library into a named, multi-collection store. Each collection has its own description used in the tool schema so Gemma can route queries to the right collection. The UI gains two-level navigation (collections → sources) and CRUD for collections.

Motivation

  • A single "MBA Textbooks" topic works while the library is coherent. Adding Economics papers creates an ambiguous mixed collection.
  • Gemma needs per-collection descriptions to route search_library calls correctly.
  • Users need to promote files from conversations or Semantic Scholar results directly into long-term library collections.
  • Move rather than re-ingest: collections as metadata fields mean moving a source is a cheap metadata update, not a re-embedding job.

Key Design Decisions

Metadata field, not separate ChromaDB collections. Each chunk gets a collection metadata field (e.g. "mba"). Moving a source between collections updates metadata only — no re-embedding. A single ChromaDB collection (collective.documents) is retained.
Single tool, collection enum parameter when >1 collection exists. When only one collection exists, the schema is identical to today (no parameter, tight description). When multiple exist, a collection string enum is added. Gemma selects the collection via the parameter — each enum value has an inline description.
Collection metadata lives in documents.yaml. The existing topic key is replaced by a collections list. ConfigManager already handles dynamic saves. No new file format needed.
Backward-compatible migration. Existing chunks without a collection field are treated as belonging to the first collection. No bulk re-tagging on startup. Users can move old sources via the UI if desired.

Config Schema Change

Current config/documents.yaml:

topic: "MBA textbooks covering strategy, finance, marketing, and operations"
chunk_size: 800
chunk_overlap: 80

New config/documents.yaml:

chunk_size: 800
chunk_overlap: 80
collections:
  - name: mba
    display_name: "MBA Textbooks"
    description: "MBA textbooks covering strategy, finance, marketing, and operations"

Migration: if collections is absent, auto-create one collection from the existing topic field. Old chunks (no collection metadata) are treated as belonging to the first collection.

Implementation Plan

12a · Data Layer

DocumentService — collection-aware CRUD

ChangeDetail
ingest_file(path, collection, …)Adds collection field to each chunk's metadata
search(query, collection=None, …)Where-filters on collection if provided; searches all if None
list_collections()Returns distinct collection values from chunk metadata
list_sources(collection=None)Optional filter; sources without collection field belong to first collection
move_source(name, to_collection)Updates collection metadata on all chunks for that source (ChromaDB update())
delete_collection(name)Deletes all chunks where collection == name
rename_collection(old, new)Updates collection metadata on all matching chunks
12b · Tool Layer

SearchLibraryTool — dynamic schema

  • On startup and on schema re-announce, reads collections from documents.yaml
  • One collection → schema identical to current (description = collection description, no collection param)
  • Multiple collections → adds collection string enum; each enum value's description is the collection description
  • Tool re-announces whenever DocumentsWindow saves a collection change (existing TOOL_SCHEMA_REQUEST mechanism)
# Single collection (unchanged from today)
{
  "name": "search_library",
  "description": "Search MBA Textbooks: MBA textbooks covering strategy, finance…",
  "parameters": { "query": {"type": "string", "description": "…"} }
}

# Multiple collections
{
  "name": "search_library",
  "description": "Search the document library. Choose the collection that best matches your query.",
  "parameters": {
    "query": {"type": "string"},
    "collection": {
      "type": "string",
      "enum": ["mba", "economics"],
      "description": "mba: MBA textbooks (strategy, finance, ops); economics: macro/micro research papers"
    }
  }
}
12c · UI Layer

DocumentsWindow — two-level navigation

  • Collections view: table of collections (display_name, source count, description snippet); ✎ rename inline, 🗑 delete with confirmation
  • "+ Collection" button: prompts for name + description in a dialog
  • Source view: click a collection row to drill in; breadcrumb header shows current collection; back button returns to collections view
  • Move to…: combo box on each source row listing other collections; selecting one calls move_source() and refreshes
  • Description bar: shown in source view below the table; editable, Save triggers schema re-announce
  • + Files / + Folder: ingest into the currently viewed collection
12d · Promote to Library

"Save to library" from attachment bar and Semantic Scholar

  • Attachment bar: each attached file gets a "→ Library" button; clicking opens a collection picker dialog and triggers background ingest
  • Semantic Scholar results: after a paper is fetched, Gemma can call a save_to_library(url_or_text, collection) tool — or a manual button in the tool activity panel
  • This phase is lower priority and can be deferred after 12a–12c
